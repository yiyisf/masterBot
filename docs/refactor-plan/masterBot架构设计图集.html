读图要点：L3 的 AgentRouter 是整个改造的「枢纽」——它把 SDK 路径与 Legacy 路径在抽象上对齐， 上层（L1/L2）和下层（L5/L6）都不需要感知底下用的是哪个 agent 引擎。这种设计让你可以渐进式迁移， 而不是一次性 rewrite。

FIGURE 02 · Hybrid Router

AgentRouter Hybrid 路由

路由层是 Hybrid 方案的灵魂。它接收一个 AgentConfig，根据 provider 字段 + FeatureFlag 决定路由到 Claude SDK 还是 Legacy 引擎。 关键设计：任何 provider 都可以强制 forceLegacy=true 走自研路径，作为兜底逃生通道。

▼ AgentConfig 入参
provider · model · sessionId · userId
forceLegacy? · resumeFromSession? · message
▼
◆ AgentRouter.route()
provider === 'anthropic'
AND
!config.forceLegacy
AND
featureFlag.enabled(userId)
▼
✓ TRUE → ClaudeManagedAgent
· @anthropic-ai/claude-agent-sdk
· 自动 caching / compaction
· 原生 fork / resume
· 内置 12 种 hooks
· Subagent context isolation
· maxTurns 250+
✗ FALSE → LegacySelfHostedAgent
· 现有 src/core/agent.ts
· OpenAI / Gemini / Ollama
· 自研 ReAct loop
· ContextManager + DAG
· 维护至 SDK 支持 OpenAI 兼容接口
· maxIterations 10 → 提升至 50

灰度策略：初期 5% 流量走 Claude SDK，对比指标稳定后逐步放量到 50% / 100%。 LegacySelfHostedAgent 永不下线——既保护多 provider 用户，也是 SDK 故障时的逃生通道。

FIGURE 03 · Hooks Pipeline

颜色语义： ● 关键安全/审批 · ● 数据保护 · ● 可观测性 · 普通 = 业务逻辑。 所有 Hook 都是独立、可测、可热插拔——你可以单独启用/停用任何一个，无需改主流程。

FIGURE 04 · Subagent Topology

Subagent 委派拓扑

Subagent 的核心价值不是"多 agent 协作"，而是context 隔离 和 模型成本梯度。 下图展示一个典型场景：用户说"帮我整理今天工作"，主 agent 自动委派给 4 个专家子 agent，每个用最合适的模型。

★ Main Agent · 主线协调者
model: claude-opus-4-7 · maxTurns: 250 · 看到 systemPrompt + AGENTS.md + 子 agent 描述
▸ secretary
个人秘书 · 综合协调日常事务 · 「整理今日待办、给出 3 件最重要的事」
tools: Task + Read sonnet-4-6
▸ email-handler
邮件助手 · 搜索/起草/发送 · 批量操作时主线只看到摘要不见数百封邮件
tools: gmail__list/get/send haiku-4-5
▸ calendar-handler
日程助手 · 查询/创建/修改/查找空闲 · 涉及外部人员强制双重确认
tools: gcal__list/create/update haiku-4-5
▸ researcher
深度研究 · 关键词扩展 + 并行抓取 + 引用结构化报告
tools: WebSearch/Fetch/Read opus-4-7

关键提升数据：子 agent 的 context 隔离机制可减少 ~67% 的跨域 token 浪费（LangChain 实测）。 模型梯度：Haiku 处理高频简单任务（邮件操作/日历查询）、Sonnet 处理协调（秘书）、Opus 处理需要深度推理的任务（研究）—— 自然形成「便宜的多、贵的少」的成本最优解。

FIGURE 05 · Memory Architecture

当前 masterBot 的短期/长期/知识图谱三套独立体系，统一为认知科学派的四层结构： Working / Episodic / Semantic / Procedural。每层各司其职、有明确生命周期、有独立写入路径。

核心改动：引入 memory_consolidate 工具 让 agent 自主决定何时压缩、归档（active context compression 模式）。 实测：相比 harness 强制压缩，可减少 22.7% token 消耗且无准确率损失。

FIGURE 06 · Permission Engine

Permission 五层评估流程

Claude SDK 的标准权限评估顺序。每一层都有明确职责、不能用单一 if-else 替代。 masterBot 当前的 sandbox 主要在第 2 层、HitL 卡片在第 5 层——补齐 1、3、4 层后才有真正的「深度防御」。

1
Hooks (PreToolUse)
运行你注册的 hook 函数 · 可 deny / modify / log · 灵活的程序化控制
→ deny / modify
或 ↓ 继续
2
Deny Rules · 静态黑名单
来自 disallowedTools + .claude/settings.json · 即使 bypassPermissions 也无法绕过
→ 强制 deny
3
Permission Mode · 全局模式
default · acceptEdits · plan · bypassPermissions · 不同用户/session 可设置不同模式
→ 直接 allow
或 → deny
4
Allow Rules · 静态白名单
来自 allowedTools 配置 · 如 ['Read', 'Glob', 'Grep'] · 命中即放行
→ allow
5
canUseTool · 运行时审批
交互式回调 · masterBot 现有的 IM 审批卡片在此层 · 支持 approve with changes（修改入参后放行）
→ allow / deny /
→ approve+modify

Lethal Trifecta 检测：当一次 session 同时启用「私密数据访问」+「不可信内容暴露」+「外部通信」三类工具时， 会强制触发第 5 层 HitL——这是 prompt injection 攻击最危险的组合。 在第 1 层 Hook 中实现该检测，早于所有其他规则评估。

FIGURE 08 · Migration Roadmap

Phase 演进路线

9 个 Phase 的渐进式迁移路径，总周期约 14 周。每个 Phase 都独立可回滚、 独立可灰度、独立交付价值。 重点：Phase 1（可观测性）必须先行，没有 trace 就没法做后续的 A/B 对比。

P0
准备工作 1 week #issue-1 ~ #issue-3
立项决策 ADR · SDK 依赖添加 · docs/migration 目录建立 · 交付：跑通官方 quickstart
P1
可观测性先行 1 week #issue-4 ~ #issue-7
引入 OTel + 替换 SpanRecorder + 部署 Langfuse · 交付：所有现有功能在 Langfuse 上有完整 trace
P2
抽象层 + Hooks 重构 2 weeks #issue-8 ~ #issue-16
IAgent 接口 · AgentRouter · Legacy 引擎重命名 · sandbox/IM/memory/PII 全部 hookify · 交付：测试 100% 通过、内部架构对齐 SDK
P3
★ ClaudeManagedAgent 上线 2 weeks #issue-17 ~ #issue-22
ClaudeManagedAgent 实现 · SDK MCP Wrapper · 灰度切换开关 · capability eval 套件 · 里程碑：5% 流量走 SDK，效果指标可对比
P4
Skills + Subagents 升级 2 weeks #issue-23 ~ #issue-27
Progressive Disclosure 改造 · 4+ Subagent 定义 · SOUL.md 迁移 · 预期：主 agent 平均 input tokens 减少 ≥30%
P5
Session 高级特性 1 week #issue-28 ~ #issue-32
数据库 schema 变更 · fork / resume / checkpoint 端点 · Web UI rewind 按钮
P6
Memory 四层重构 2 weeks #issue-33 ~ #issue-37
MemoryRouter 抽象 · pgvector 升级 · Procedural Memory · active compression · HitL 写入门
P7
Multi-Channel 个人化 2 weeks #issue-38 ~ #issue-41
Telegram · iMessage · IChannel 抽象 · 个人化技能 (calendar/email/note sync)
P8
前端升级 1 week #issue-42 ~ #issue-43
AG-UI 协议替换 assistant-runtime · CopilotKit 集成评估
P9
评估体系 持续 #issue-44 ~ #issue-47
promptfoo 集成 · GitHub Actions eval · Langfuse Datasets · LLM-as-judge

关键依赖：P1 → P2 → P3 是必走链路（先看见，再抽象，再切换）。P4-P9 大部分可并行推进。 风险预警：P3 是技术风险最高的一步——务必从 5% 流量起步、Langfuse 实时对比指标后再放量。
